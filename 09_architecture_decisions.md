# Architecture Decisions

This section documents the most important architectural decisions that have a significant impact on the structure, maintainability, and long-term evolution of the University Management System. Each decision was made after considering alternative approaches and evaluating them against the project's [requirements](01_introduction_and_goals.md) and [constraints](02_architecture_constraints.md).

| Problem                                                                                                                              | Considered Alternatives                                                 | Decision                                                                                                                                                                                                                                         |
|--------------------------------------------------------------------------------------------------------------------------------------|-------------------------------------------------------------------------|--------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| The system should support multiple university processes within a single application while remaining maintainable and easy to extend. | Microservices, Service-Oriented Architecture (SOA), Modular Monolith    | A modular monolith was selected because the business domains are closely related and developed by a relatively small team. This approach provides clear separation of concerns while avoiding the operational complexity of distributed systems. |
| The application must exchange data with the existing Student Information System (SIS) and Financial Management System (FMS).         | Direct database access, file import/export, REST API integration        | REST API integration was chosen because it provides a loosely coupled and well-defined communication interface. It simplifies maintenance and allows external systems to evolve independently.                                                   |
| Different user groups require different permissions and access rights.                                                               | Shared access, manually assigned permissions, role-based access control | Role-Based Access Control (RBAC) was chosen because it offers a clear and scalable way to manage permissions for students, lecturers, administrators, and finance staff while supporting the security requirements of the system.                |
